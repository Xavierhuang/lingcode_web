工作流

多 Claude Code 会话

同时运行多个 Claude Code 标签。让重构、写文档、调试并行进行——Cursor 没有这项能力。

为什么多会话重要

只有一个 Claude Code 标签时，你在等。开了一次重构，就只能坐着看 AI 思考。想同时写文档？不行。想在第一个会话跑着的时候调试另一个模块？切换标签就丢上下文。

LingCode 给你多个 Claude Code 会话。每个标签跑自己的 claude 进程。在标签 1 重构、标签 2 写文档、标签 3 调试——同时进行。不切换、不等待。

如何开一个新会话

在任意标签上右键，选择 Claude Code：新会话。或从菜单走：Claude Code → 新会话。新标签会有它自己的 Claude Code 实例。每个标签保留自己的对话和状态。

加速你的工作流

重构 + 文档并行

标签 1："把这个模块重构成 async/await。" 标签 2："为认证流程写 README 章节。" 在 Claude 忙着重构时，写文档的会话已经在跑了。两边输出都有，谁也不阻塞谁。

调试 + 新功能

标签 1："这个 API 为什么返回 500？" 标签 2："给用户偏好设置加个新接口。" 一个会话里排查 Bug，另一个里开发下一个功能。不必为一件事停下另一件。

审查 + 修复 + 规划

标签 1："对支付模块执行 /review。" 标签 2："修复审查中 auth.ts 的问题。" 标签 3："起草数据库 Schema 的迁移方案。" 三股工作流，一个开发者——交付更快。

探索 + 实现

标签 1："在这套技术栈里实现实时同步的最佳方案是什么？" 标签 2："实现我们刚讨论过的登录流程。" 用一个会话当研究助手，另一个负责执行。上下文不再来回切换。

并行会话的小贴士

  • 任务聚焦。 每个会话一件主要任务。一个标签里塞太多议题会拖慢 Claude。
  • 明示意图。 每个会话都用清晰的 Prompt 开头。"重构 X"和"记录 Y"比模糊的"帮我一下"好用得多。
  • 留意 API 用量。 每个会话分别计费。只在时间节省值回成本时再开多会话。
  • 完事就关。 会话完成了？关掉标签。LingCode 会让其他会话继续跑。不必囤标签。

为什么 LingCode 能做到

LingCode 由 Swift 构建。每个 Claude Code 标签都是一个跑 claude CLI 的真实终端。原生架构让我们能同时保活多个进程，不用承担基于浏览器的编辑器那种开销。Cursor 用的是一个自研的 Claude Code UI；我们跑的是真的 CLI，你需要几个就跑几个。
