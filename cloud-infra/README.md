# LingCode Cloud — data plane (Phase 1)

LingCode-operated backend infrastructure for /try prototypes. This is the
**data plane**: a shared Postgres cluster where each saved prototype gets an
isolated schema (`be_<backend_id>`) + a per-tenant Postgres role. The
**control plane** is the existing Express app (`website/server/`), which
connects here as `cloud_admin` to provision and to proxy tenant data.

> This stack runs on LingCode-operated hosts only. **Do not** run it on the
> lingcode.dev marketing droplet (~8.7G) — use a separate, sized host.

## Run it (local)

```bash
cd website/cloud-infra
docker compose up -d
# wait for postgres healthcheck, then:
psql "postgres://cloud_admin:devpass@localhost:5544/lingcloud" -c '\dn'
```

Point the control plane at it (env for `website/server`):

```bash
export CLOUD_PG_ADMIN_URL="postgres://cloud_admin:devpass@localhost:5544/lingcloud"
export CLOUD_JWT_SECRET="$(openssl rand -hex 32)"   # >= 32 chars
export CLOUD_POSTGREST_URL="http://localhost:5545"  # Phase 2 gateway base (unused in Phase 1)
```

If these env vars are unset, the Cloud console reports "not configured" and
the rest of /try is unaffected.

## Isolation model

- Each tenant = one Postgres **schema** `be_<backend_id>` + one **role**
  `trole_<backend_id>` (NOLOGIN) granted `USAGE` on that schema only, plus
  table privileges via `ALTER DEFAULT PRIVILEGES`.
- The per-tenant **anon JWT** (minted by the control plane, signed with
  `CLOUD_JWT_SECRET`) carries `{ "role": "trole_<backend_id>" }`. Any data
  request runs `SET LOCAL ROLE` to that role, so it can only reach that
  tenant's schema. RLS can be layered on top per table.
- The SQL editor + table viewer go through the **control plane** as
  `cloud_admin` with `search_path` pinned to the tenant schema — never
  through the tenant's public credentials.

## Phase 1 vs Phase 2 (honest)

**Phase 1 (this slice):** generated apps read/write through the control-plane
data proxy at `POST /api/cloud/be/:backendId/select|insert`, authenticated by
the anon JWT. This avoids PostgREST's dynamic-multi-schema config reload and
keeps the slice runnable end-to-end. PostgREST is in `docker-compose.yml` but
off the data path.

**Phase 2:** stand up a subdomain gateway
(`<backend_id>.api.lingcode.dev`) in front of PostgREST, wire dynamic
`db-schemas` reload on provision, and swap the injected client from the proxy
to PostgREST. The control-plane API surface stays the same.

## Connection pooling (PgBouncer)

Production runs **PgBouncer in transaction mode** between the control plane and
Postgres (`Node → PgBouncer :6432 → Postgres :5544`). The control plane connects
to PgBouncer, not Postgres directly — set `CLOUD_PG_ADMIN_URL` to the `:6432`
port (`bootstrap-remote.sh` prints the exact line).

Transaction pooling is safe here because every tenant query wraps work in
`BEGIN … SET LOCAL ROLE … SET LOCAL search_path … set_config(…, true) … COMMIT`
— all session state is transaction-local and resets at COMMIT, and
node-postgres uses unnamed prepared statements. There is no plain `SET`, named
prepared statement, session advisory lock, or `LISTEN/NOTIFY` on the hot path,
so a server connection is safe to recycle at each COMMIT. **If you ever add a
session-pinned path, give it a `pool_mode=session` database entry in
`pgbouncer/pgbouncer.ini` rather than changing the default.**

Config is in [`pgbouncer/pgbouncer.ini`](pgbouncer/pgbouncer.ini); auth uses a
SCRAM verifier in `pgbouncer/userlist.txt` (gitignored, generated by the
bootstrap from `pg_authid` — the plaintext password never lands in the pooler
config). Inspect live pools: `docker exec lingcloud-pgbouncer psql -p 6432 -U
cloud_admin pgbouncer -c 'SHOW POOLS;'`.

Pooling removes the connection ceiling that blocked running >1 API process, and
realtime is now cross-process safe (see below), so the API can be scaled out.

This is a connection-string flip to roll back: point `CLOUD_PG_ADMIN_URL` back
at `:5544` (direct Postgres) and restart `lingcode-api`.

### Realtime is cross-process (LISTEN/NOTIFY)

`emitChange()` in `cloud-data-plane.js` no longer fans out in-process — it issues
`pg_notify('lingcloud_change', …)`, and a dedicated LISTEN connection in **every**
API process re-emits onto that process's local `realtimeBus`. So a write handled
by process A reaches SSE subscribers on process B. SSE route logic is unchanged
(it still subscribes to `realtimeBus`).

**`CLOUD_PG_DIRECT_URL` is required in prod.** The LISTEN connection must bypass
PgBouncer — transaction pooling silently breaks `LISTEN` — so it connects
straight to Postgres `:5544`. `CLOUD_PG_ADMIN_URL` (pooled, `:6432`) is still used
for everything else, including issuing the `NOTIFY`. In local dev, where
`CLOUD_PG_ADMIN_URL` already points directly at Postgres, `CLOUD_PG_DIRECT_URL`
can be omitted (the listener falls back to it). Rows larger than ~7.9 KB exceed
the NOTIFY payload cap and are delivered body-less with an `oversized` flag
(owner subscribers get a refetch signal; RLS subscribers drop them).

## Teardown

```bash
docker compose down        # keep data
docker compose down -v      # also drop the volume (wipes all tenant schemas)
```

## Backups & PITR (.228)

The recovery system is **pgBackRest** with point-in-time recovery: physical base
backups + continuously-archived WAL, in an **encrypted, off-host** repository on
DigitalOcean Spaces. This is what survives a VM/disk loss and lets you restore
to any second, not just to a nightly snapshot.

How it's wired:

- The Postgres image ([`Dockerfile.postgres`](Dockerfile.postgres)) ships the
  `pgbackrest` binary so `archive_command` can run inside the container.
- WAL archiving GUCs (`archive_mode=on`, `archive_command='pgbackrest …
  archive-push %p'`, etc.) are applied as explicit `-c` flags in the postgres
  service of [`docker-compose.prod.yml`](docker-compose.prod.yml) — documented,
  with rationale, in [`postgres-conf.d/10-archive.conf`](postgres-conf.d/10-archive.conf)
  (reference; `include_dir` isn't settable via `postgres -c`). Note `archive_mode`
  is **not reloadable** — it takes effect on a Postgres restart, which the
  bootstrap handles by recreating the container.
- Repo config is [`pgbackrest/pgbackrest.conf`](pgbackrest/pgbackrest.conf.example)
  (gitignored; rendered from the `.example` by the bootstrap). It targets Spaces
  (`repo1-type=s3`), encrypts at rest (`repo1-cipher-type=aes-256-cbc`, pass in
  `.env`), and keeps `repo1-retention-full=4`.

Enable it by putting `SPACES_KEY` / `SPACES_SECRET` in `.env` and running
`bootstrap-remote.sh` (it does `stanza-create`, `check`, and a first full
backup). Cron, installed by the bootstrap, runs on the cloud-pg host
(`138.197.107.228`) writing logs to `/opt/lingcloud-backups/`:

```
17 3 * * *   pgbackrest-backup.sh        # base backup -> Spaces (full Sun / diff otherwise)
0  5 * * 0   pgbackrest-restore-test.sh  # WEEKLY restore drill into a throwaway container
30 4 * * *   backup-cloud-pg.sh          # logical pg_dump export (portable fallback, local)
*/30 * * * * disk-alert.sh               # warn at >= 85% disk (set ALERT_WEBHOOK to get pinged)
```

**Restore to a point in time** (e.g. just before a bad migration):

```bash
# Stop Postgres, then restore into its data dir and replay WAL to the target:
docker exec lingcloud-postgres pgbackrest --stanza=lingcloud \
  --type=time --target='2026-06-03 14:00:00' --target-action=promote restore
# Start Postgres; it recovers to the target and promotes.
```

[`pgbackrest-restore-test.sh`](pgbackrest-restore-test.sh) proves backups are
restorable on a schedule (restores into a throwaway container, replays WAL, runs
a sanity query, tears down) — an unrestored backup is a hypothesis, not a safety
net. Set `PITR_TARGET="<timestamp>"` to drill a specific point. Repo health any
time: `docker exec lingcloud-postgres pgbackrest --stanza=lingcloud check`.

### Logical export (portable fallback)

`backup-cloud-pg.sh` still runs nightly as a **portable logical export** (good
for schema-level or cross-version restores), but it is no longer the recovery
system. Tenant data is per-schema (`be_<id>`) owned by per-tenant roles
(`trole_<id>`), so it dumps **both** `globals-*.sql.gz` (roles/grants) and
`lingcloud-*.dump` (full DB), writes to `.partial` then renames, and refuses to
run past 92% disk. Restore on a fresh container:

```bash
gunzip -c globals-<stamp>.sql.gz | docker exec -i lingcloud-postgres psql -U cloud_admin -d postgres
docker exec -i lingcloud-postgres pg_restore -U cloud_admin -d lingcloud --clean --if-exists < lingcloud-<stamp>.dump
```

Inspect a dump without restoring: `docker exec -i lingcloud-postgres pg_restore --list < <dump>`.

## High availability — warm standby (manual promote)

PITR means you won't *lose* data; a warm standby means a primary VM death is a
**promote (seconds–minutes)** instead of a restore (minutes–hours), and gives
read-scaling headroom. This is **physical streaming replication** (not logical):
it copies the whole cluster byte-for-byte — roles, schemas, DDL, every tenant —
which is exactly what the schema-per-tenant + per-tenant-role model needs.

Prereqs are already set by the prod overlay's `-c` flags (values documented in
[`postgres-conf.d/10-archive.conf`](postgres-conf.d/10-archive.conf)):
`wal_level=replica`, `max_wal_senders=3`, and `max_slot_wal_keep_size` (so an
unconsumed replication slot can't fill the primary's disk — re-seed the standby
if its slot is invalidated).

**Topology.** Each data-plane VM is a self-contained `{Postgres + PgBouncer}`
unit running the identical stack. A **DigitalOcean reserved IP** points at the
live VM; the API's `CLOUD_PG_ADMIN_URL` (`:6432`) and `CLOUD_PG_DIRECT_URL`
(`:5544`) target the reserved IP. Failover = remap the reserved IP — **no API
change**. The standby is read-only until promoted.

### Set up

```bash
# 1) On the PRIMARY — authorize a standby and create the slot:
STANDBY_IP=<standby private IP> sudo bash ha/ha-setup-primary.sh

# 2) On the STANDBY VM (repo rsync'd here; .env carries the SAME
#    PGBACKREST_CIPHER_PASS + Spaces creds + REPLICATION_PASSWORD as the primary):
PRIMARY_IP=<primary private IP> sudo bash ha/ha-seed-standby.sh
```

`ha-seed-standby.sh` seeds from the encrypted pgBackRest repo
(`pgbackrest --type=standby restore`) — **zero base-backup load on the live
primary** — and starts it streaming. Check health on either node:

```bash
bash ha/ha-status.sh   # PRIMARY: shows connected standbys + lag; STANDBY: receive/replay lag
```

### Failover (promote)

```bash
# On the STANDBY, when the primary is lost (or for a planned switch):
sudo bash ha/ha-promote.sh
```

Then, as the script reminds you: **(1)** remap the reserved IP to the promoted
VM (`doctl compute reserved-ip-action assign …`), and **(2) fence the old
primary** (power off / firewall) so it can't come back as a second primary
writing to the same Spaces stanza — split-brain corrupts both the data and the
backup repo. The promoted node archives WAL on the new timeline automatically;
verify with `pgbackrest --stanza=lingcloud check`. Rebuild the old primary as a
fresh standby of the new primary (`ha-setup-primary.sh` on the new primary →
`ha-seed-standby.sh` on the rebuilt node).

### Honest limits

- **Manual promote** = a human in the loop and a short outage window. That's the
  deliberate trade for not running Patroni/repmgr (etcd/consul + a daemon) on a
  two-node setup. Add automation later if uptime SLAs demand it.
- **Read scaling** is a ready follow-on: `_asTenant` already takes a `readOnly`
  flag, so read-only traffic (SQL editor, advisors, introspection) can be routed
  to a second pool pointed at the replica without re-architecting. Defer until
  the standby is proven.
- **Buy-vs-build:** DigitalOcean Managed Postgres bundles HA standby + PITR +
  PgBouncer as a service. If the goal is an uptime SLA rather than infra
  ownership, that collapses pooling + PITR + HA into one managed product — worth
  weighing against operating this yourself.

## Observability — Prometheus + Grafana

Two layers feed one Prometheus/Grafana stack:

**App metrics (the API box).** The Node API exposes Prometheus metrics at
token-gated `GET /metrics` (`website/server/metrics.js`): HTTP request rate /
latency / errors (labelled by route *pattern*, never raw ids), pg pool
gauges, and a cloud-query duration histogram. `GET /api/health/deep` probes the
DB (`SELECT 1` + pool stats; 503 on failure) vs the static `/api/health`
liveness. Owner mutations now write `source='audit'` events to `backend_logs`
(filter them in the console **Logs** tab). Set `METRICS_TOKEN` in the API box
`.env`, and expose `/metrics` to the monitoring host only — e.g. on the API
box's private nginx vhost:

```
location = /metrics { allow <monitoring-ip>; deny all; proxy_pass http://127.0.0.1:3000/metrics; }
```

**Data-plane exporters (the PG VM).** The prod overlay runs `postgres_exporter`
(as the least-privilege `monitoring` role), `pgbouncer_exporter`, and
`node_exporter`, all bound to the private IP. `pg_stat_statements` is enabled by
`shared_preload_libraries` (a `-c` flag in [`docker-compose.prod.yml`](docker-compose.prod.yml);
documented in [`postgres-conf.d/20-observability.conf`](postgres-conf.d/20-observability.conf)
— **needs the same restart as `archive_mode`**); the
extension + monitoring role are created by
[`observability/obs-setup-db.sh`](observability/obs-setup-db.sh) (run by the
bootstrap). Firewall ports 9187/9127/9100 to the monitoring host only.

**Monitoring host (separate VM or the API box).** Prometheus + Alertmanager +
Grafana, all private-bound:

```bash
# put DATA_PLANE_IP, API_IP, ALERT_WEBHOOK, METRICS_TOKEN, GRAFANA_ADMIN_PASSWORD in .env, then:
bash observability/bootstrap-monitoring.sh
# reach the UIs by SSH tunnel only — nothing is published publicly:
ssh -L 3001:127.0.0.1:3001 -L 9090:127.0.0.1:9090 <monitoring-host>
#   Grafana http://localhost:3001  ·  Prometheus http://localhost:9090/targets
```

Alerts ([`observability/alerts.yml`](observability/alerts.yml)) route through
Alertmanager to the **same `ALERT_WEBHOOK`** the cron uses (one channel): targets
down, connections near max, replication slot inactive / pinning WAL, PgBouncer
clients waiting, disk < 15%, API 5xx-rate / p95-latency. Grafana auto-loads the
**LingCode Cloud** dashboard (provisioned).

> ⚠️ Exporter metric **names vary by version**. After first scrape, confirm each
> alert's metric exists on the live exporter (Prometheus → Graph, or curl the
> exporter `/metrics`) — a rule on a missing metric silently never fires.
> Backup-age/restore alerts remain on the cron webhook, not Prometheus.

**Verify:** `obs-setup-db.sh` then `SELECT count(*) FROM pg_stat_statements`
works; Prometheus `/targets` shows every job **up**; the Grafana dashboard renders
live data; stop the standby and confirm `ReplicationSlotInactive` reaches the
webhook; confirm exporter/Grafana ports are refused from outside the VPC.
